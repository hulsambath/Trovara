# Vector Storage & Search (RAG Step 2)

> Stores note embeddings in ObjectBox and performs cosine-similarity search
> to find the most relevant note chunks for a user query.

This document covers the **Retriever** layer of Trovara's RAG pipeline — the
component responsible for persisting vector embeddings and searching them at
query time.

## 1. Overview

Trovara stores each note's content as vector embeddings (generated by
`EmbeddingService` in Step 1). The **Vector Storage & Search** layer
provides:

- **Persistence** — embeddings saved in ObjectBox via `IEmbeddingRepository`
- **Retrieval** — brute-force cosine-similarity search via `VectorSearchService`
- **Ranking** — results sorted by descending similarity, filtered by threshold

This is the bridge between embedding generation (Step 1) and document
resolution / prompt augmentation (Steps 3–5).

```
EmbeddingService                VectorSearchService
(Step 1)                        (Step 2 — this layer)
     │                               │
     │  embedNote() ──► save ──►     │
     │                          IEmbeddingRepository
     │  embedQuery() ─────────►      │
     │                          search(queryVector)
     │                               │
     │                          ┌────┴────┐
     │                          │ Top-K   │
     │                          │ Scored  │
     │                          │ Chunks  │
     │                          └─────────┘
```

## 2. Architecture

### Data flow

```
User Query (string)
    │
    ▼
EmbeddingService.embedQuery()      → List<double> (768-dim vector)
    │
    ▼
VectorSearchService.search()       → List<ScoredEmbedding>
    │
    ├─ Load all embeddings from IEmbeddingRepository
    ├─ Compute cosine similarity for each
    ├─ Filter by minScore threshold
    ├─ Sort by descending score
    └─ Return top-K results
```

### Design decisions

| Decision                  | Rationale                                                                                        |
| ------------------------- | ------------------------------------------------------------------------------------------------ |
| Brute-force search        | ObjectBox Dart has no HNSW/ANN index support; brute-force is fast enough for < 10K chunks        |
| In-memory computation     | All embeddings loaded from ObjectBox into memory for search; acceptable for personal note counts |
| Cosine similarity         | Standard metric for text embeddings; Gemini embeddings are normalized, so cosine ≈ dot product   |
| Serialized string storage | Embedding vectors stored as comma-separated strings since ObjectBox lacks float-list columns     |
| Repository pattern        | `IEmbeddingRepository` interface allows swapping storage backends (ObjectBox, SQLite, etc.)      |

## 4. Embedding Repository

### Interface — `IEmbeddingRepository`

**File:** `lib/core/repository/interfaces/embedding_repository.dart`

Defines the contract for embedding storage. Follows the Interface Segregation
Principle — only embedding-related operations.

| Method                          | Description                                         |
| ------------------------------- | --------------------------------------------------- |
| `initialize()`                  | Opens the ObjectBox box                             |
| `saveEmbedding(embedding)`      | Persist a single `NoteEmbedding`                    |
| `saveEmbeddings(list)`          | Batch save multiple embeddings                      |
| `getEmbeddingsByNoteId(noteId)` | Fetch all chunks for a specific note                |
| `getAllEmbeddings()`            | Load all embeddings (used for brute-force search)   |
| `deleteByNoteId(noteId)`        | Remove all chunks for a note (on permanent delete)  |
| `deleteAll()`                   | Wipe all embeddings (e.g. on model version upgrade) |
| `totalEmbeddings`               | Count of stored embedding entries                   |
| `dispose()`                     | Clean up resources                                  |

### Implementation — `ObjectBoxEmbeddingRepository`

**File:** `lib/core/repository/implementations/objectbox_embedding_repository.dart`

Uses the shared `ObjectBoxStoreManager` to access the `NoteEmbedding` box.
Queries use the generated `NoteEmbedding_` class for type-safe field access.

## 5. Vector Search Service

### `VectorSearchService`

**File:** `lib/core/services/vector_search_service.dart`

The core retrieval component. Loads all embeddings from the repository and
performs brute-force cosine similarity search against a query vector.

### Public API

#### `search(queryEmbedding, {topK, minScore})`

Find the top-K most similar chunks to the query embedding.

```dart
final results = vectorSearchService.search(
  queryVector,
  topK: 5,       // max results (default: 5)
  minScore: 0.3, // similarity threshold (default: 0.3)
);

for (final result in results) {
  print('Note ${result.embedding.noteId}: ${result.score}');
  print('Text: ${result.embedding.chunkText}');
}
```

**Parameters:**

| Parameter        | Type           | Default | Description                         |
| ---------------- | -------------- | ------- | ----------------------------------- |
| `queryEmbedding` | `List<double>` | —       | 768-dim vector from `embedQuery()`  |
| `topK`           | `int`          | 5       | Maximum results to return           |
| `minScore`       | `double`       | 0.3     | Minimum cosine similarity threshold |

**Returns:** `List<ScoredEmbedding>` sorted by descending score.

#### `searchExcluding(queryEmbedding, {excludeNoteIds, topK, minScore})`

Same as `search()` but skips embeddings from the specified note IDs. Useful
for finding related notes while excluding the note the user is currently
viewing.

```dart
final results = vectorSearchService.searchExcluding(
  queryVector,
  excludeNoteIds: {currentNoteId},
  topK: 5,
);
```

#### `findSimilarToNote(noteId, {topK, minScore})`

Find notes similar to a given note's content. Averages all chunk embeddings
for the source note, then searches for similar chunks (excluding the source
note itself).

```dart
// "Related notes" feature
final similar = vectorSearchService.findSimilarToNote(
  noteId,
  topK: 3,
  minScore: 0.5,
);
```

**Behavior:**

1. Fetches all embedding chunks for the source note
2. Computes element-wise average of all chunk vectors
3. Calls `searchExcluding()` with the averaged vector
4. Returns empty list if the note has no embeddings

#### `getStats()`

Returns an `EmbeddingStats` object with storage statistics.

```dart
final stats = vectorSearchService.getStats();
print(stats);
// EmbeddingStats(totalChunks: 150, uniqueNotes: 80, avgChunksPerNote: 1.88, dimension: 768)
```

### Cosine Similarity

The similarity function computes the standard cosine similarity:

$$\text{cosine\_similarity}(\mathbf{a}, \mathbf{b}) = \frac{\mathbf{a} \cdot \mathbf{b}}{|\mathbf{a}| \times |\mathbf{b}|}$$

Where:

- $\mathbf{a} \cdot \mathbf{b}$ is the dot product
- $|\mathbf{a}|$ and $|\mathbf{b}|$ are the L2 norms

**Properties:**

- Returns values in $[-1, 1]$
- Gemini embeddings are normalized, so values typically fall in $[0, 1]$
- Returns `0.0` for mismatched dimensions or zero-magnitude vectors

### `ScoredEmbedding`

A simple data class that pairs a `NoteEmbedding` with its similarity score:

| Field       | Type            | Description                 |
| ----------- | --------------- | --------------------------- |
| `embedding` | `NoteEmbedding` | The matched embedding chunk |
| `score`     | `double`        | Cosine similarity score     |

### `EmbeddingStats`

Read-only statistics about the embedding store:

| Field                | Type     | Description                                   |
| -------------------- | -------- | --------------------------------------------- |
| `totalChunks`        | `int`    | Total embedding entries stored                |
| `uniqueNotes`        | `int`    | Number of distinct notes with embeddings      |
| `avgChunksPerNote`   | `double` | Average chunks per note                       |
| `embeddingDimension` | `int`    | Vector dimension (768 for text-embedding-004) |

## 6. Dependency Injection

`VectorSearchService` is registered in the `ServiceLocator` with lazy
initialization. It depends on `IEmbeddingRepository` (shared with
`EmbeddingService`).

```
ServiceLocator
    │
    ├── embeddingRepository  ─── ObjectBoxEmbeddingRepository
    │        │
    │        ├──── used by ──── EmbeddingService  (Step 1)
    │        │
    │        └──── used by ──── VectorSearchService  (Step 2)
    │
    └── vectorSearchService  ─── VectorSearchService
```

**Access:**

```dart
final vectorSearchService = ServiceLocator().vectorSearchService;
```

The service requires no explicit initialization — it uses the repository
that's already initialized by `EmbeddingService` during app startup.

## 7. Usage Examples

### Basic RAG query (Steps 1 + 2 combined)

```dart
final sl = ServiceLocator();

// Step 1: Embed the user's question
final queryVector = await sl.embeddingService.embedQuery(
  'What did I write about meditation?',
);

if (queryVector == null) {
  print('Failed to generate query embedding');
  return;
}

// Step 2: Search for similar note chunks
final results = sl.vectorSearchService.search(
  queryVector,
  topK: 5,
  minScore: 0.3,
);

// Display results
for (final result in results) {
  print('─────────────────────────');
  print('Note ID: ${result.embedding.noteId}');
  print('Score:   ${result.score.toStringAsFixed(4)}');
  print('Chunk:   ${result.embedding.chunkText.substring(0, 100)}...');
}
```

### Find related notes

```dart
// Show "Related Notes" in the note detail view
final similar = sl.vectorSearchService.findSimilarToNote(
  currentNote.id,
  topK: 3,
  minScore: 0.5,
);

if (similar.isNotEmpty) {
  print('Related notes:');
  for (final s in similar) {
    print('  • Note ${s.embedding.noteId} (${(s.score * 100).toInt()}% match)');
  }
}
```

### Monitor embedding store

```dart
final stats = sl.vectorSearchService.getStats();

print('Embedded ${stats.uniqueNotes} notes '
    '(${stats.totalChunks} chunks, '
    '${stats.avgChunksPerNote.toStringAsFixed(1)} chunks/note, '
    '${stats.embeddingDimension}-dim vectors)');
```

## 8. Performance

### Search complexity

- **Time:** $O(n \times d)$ where $n$ = number of stored chunks, $d$ = embedding dimension (768)
- **Space:** $O(n \times d)$ for loading all embeddings into memory

### Benchmarks (estimated)

| Notes  | Chunks (avg 1.2/note) | Embedding Data | Search Time | Memory |
| ------ | --------------------- | -------------- | ----------- | ------ |
| 100    | ~120                  | ~0.4 MB        | ~1–2 ms     | ~1 MB  |
| 500    | ~600                  | ~1.8 MB        | ~2–5 ms     | ~4 MB  |
| 2,000  | ~2,500                | ~7.5 MB        | ~5–15 ms    | ~15 MB |
| 5,000  | ~6,000                | ~18 MB         | ~15–30 ms   | ~35 MB |
| 10,000 | ~12,000               | ~36 MB         | ~30–60 ms   | ~70 MB |

### Bottleneck: deserialization

The largest cost is deserializing embedding strings (`"0.123,0.456,..."`)
back into `List<double>`. This happens on every `getAllEmbeddings()` call.
See [Future Improvements](#10-future-improvements) for optimization options.

## 10. Future Improvements

### Short-term

- **Cached deserialization** — Cache `List<double>` in a `late` field on
  `NoteEmbedding` to avoid repeated string-to-double parsing
- **Isolate offloading** — Move search to `Isolate.run()` for collections
  > 5,000 chunks to keep the UI thread free
- **Pre-filtered search** — Add `searchByFolder()` or `searchByTags()` to
  narrow the search space before computing similarity

### Medium-term

- **Float64List storage** — Store embeddings as typed arrays for
  SIMD-friendly memory access and faster dot-product computation
- **Incremental index** — Maintain an in-memory cache of deserialized
  embeddings, updated on note save/delete, instead of re-loading from
  ObjectBox on every search
- **Batch query** — Allow multiple queries in one call to amortize the
  cost of loading all embeddings

### Long-term

- **ANN index** — Approximate Nearest Neighbor search (e.g. HNSW) if
  ObjectBox adds support or via a native FFI library
- **External vector DB** — Cloud-hosted Pinecone/Qdrant for collections
  exceeding 50,000 chunks
- **Hybrid search** — Combine vector similarity with keyword matching
  (BM25) for better retrieval quality
