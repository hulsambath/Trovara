# /audit — Quality Audit Orchestrator

## Flags

- (none) — default mode: run the 4 promoted commands in parallel
- `--full` — also dispatch 5 inline one-shot agents (bugs, fraud, bias, neg-correlation, perf)
- `--only <name>` — run a single sub-command; valid names: `security`, `coverage`, `architecture`, `deps`, `bugs`, `fraud`, `bias`, `neg-correlation`, `perf`

## Severity / Effort scale

- **Severity:** `Critical` (data loss / breach) · `High` (crash / wrong-result) · `Medium` (degraded UX / tech debt with blast radius) · `Low` (style / nit)
- **Effort:** `S` (<1h) · `M` (1-4h) · `L` (>4h or design needed)

## Steps

### 1. Setup

Resolve today's date and create the audit folder:
```bash
TODAY=$(date +%Y-%m-%d)
mkdir -p "docs/superpowers/audits/$TODAY"
echo "Audit folder: docs/superpowers/audits/$TODAY"
```

### 2. Parse the flag

- If `--only <name>` was passed, jump to step 3 and dispatch only that one agent.
- If `--full` was passed, set `RUN_FULL=true`; default is `RUN_FULL=false`.

### 3. Dispatch promoted commands in parallel

In a single Agent tool batch, dispatch all 4 promoted subagents (NOT slash commands — invoke the subagents directly so we can run in parallel; the slash-command files are for ad-hoc per-dimension runs by humans):

- `security-auditor` — "Audit `lib/` for security issues per your standing instructions. Return only the table."
- `test-gaps-auditor` — "Map untested services/repos/ViewModels per your standing instructions. Return only the table."
- `architecture-auditor` — "Scan `lib/` per your standing instructions. Return only the table."
- `deps-auditor` — "Audit pubspec per your standing instructions. Return only the table."

Save each returned table to the corresponding file:
- `01_security.md`
- `03_test_gaps.md`
- `04_architecture.md`
- `05_perf_deps.md` (deps half — perf will be appended in step 4 if `--full`)

Each file is prefaced with:
```markdown
# <Title> — <date>

Generated by /audit. See `00_report.md` for aggregated view.

<table>
```

### 4. If --full, dispatch the 5 one-shot agents in parallel

In a single Agent tool batch using `subagent_type: "general-purpose"`, dispatch 5 inline agents with the prompts below. Save each table under the appropriate filename.

#### 4a. bugs-crashes-auditor → `02_bugs.md`

Prompt:

> You are the Trovara bugs/crashes auditor. Read-only. Read `CLAUDE.md`, `lib/core/CLAUDE.md`, `lib/core/services/ai/CLAUDE.md` first. Then scan `lib/` for:
>
> 1. `try/catch` blocks that swallow errors silently (no log, no rethrow, no user feedback). Grep: `catch \(.*\)\s*\{\s*\}` and `catch \(.*\)\s*\{\s*//`.
> 2. Unawaited futures: `grep -rEn "(repository|service|client)\.[a-zA-Z]+\(" lib/ | grep -v await` — manually filter for actual misses, not assignment patterns.
> 3. Null-bang `!` on nullable types in business logic: `grep -rEn "\.[a-zA-Z]+\!" lib/core lib/views`.
> 4. Stream subscriptions never cancelled (StreamSubscription stored in field but no `dispose()` or `cancel()` in `dispose`/`tearDown`).
> 5. Race conditions in sync: `lib/core/services/sync/google_drive_sync_service.dart` and `lib/core/services/chat/chat_drive_sync_service.dart` — concurrent reads/writes without locks, mutable shared state.
>
> Return only this table (no preamble):
>
> ```
> | Severity | Category | File:Line | Finding | Recommendation | Effort |
> ```
>
> Categories: `Bug`, `Crash`, `Race`, `Leak`. Severity: `Critical` (data loss), `High` (crash/wrong-result), `Medium` (degraded), `Low` (nit). Effort: `S`/`M`/`L`. Cite exact `file:line`. Don't speculate.

#### 4b. fraud-auditor → `06_fraud.md`

Prompt:

> You are the Trovara fraud/abuse auditor. Read-only. Read `CLAUDE.md`, `lib/core/CLAUDE.md`, `lib/core/services/ai/CLAUDE.md`, `lib/core/services/auth/`, `lib/constants/config_constants.dart`. Then scan for:
>
> 1. **API/cost abuse:** Unbounded RAG calls (RagService.query without a per-session/per-user rate limit), embedding loops in `embedding_service.dart` without backoff, API keys extractable from the APK (any `String.fromEnvironment` default that is a real key).
> 2. **Auth/identity abuse:** Anonymous→user-id migration in `note.dart` and `chat_thread_entity.dart` — paths where notes can lose their userId or be claimed by another user. Drive OAuth tokens stored where another app instance can read them.
> 3. **Content abuse:** Imported notes (Obsidian/Notion/Storypad adapters in `lib/core/import/adapters/`) flowing into LLM prompts unsanitized — prompt-injection payloads inside note bodies. Chat input concatenated into system prompts in `prompt_builder_service.dart`.
> 4. **Data exfiltration:** Vector search in `vector_search_service.dart` returning chunks from other users' notes. Embeddings written without userId scoping. Drive folder access wider than the current user's scope.
>
> Return only the table format above. Categories: `APIAbuse`, `AuthAbuse`, `ContentAbuse`, `Exfil`. Severity: `Critical` for confirmed cross-user leaks; `High` for unbounded cost paths or auth gaps; `Medium` for unsanitized inputs. Effort: `S`/`M`/`L`.

#### 4c. bias-auditor → `07_bias.md`

Prompt:

> You are the Trovara bias auditor. Read-only. Read `CLAUDE.md`, `lib/core/services/ai/CLAUDE.md`, `lib/core/services/ai/vector_search_service.dart`, `lib/core/services/ai/prompt_builder_service.dart`, `lib/core/services/ai/rag_service.dart`, `lib/views/insights/`. Then evaluate:
>
> 1. **RAG retrieval bias:** Does `vector_search_service.dart` favor recent notes (recency in scoring), longer notes (chunk count weighting), or English-only embeddings (km text gets weaker matches)? Read the score computation and weighting code.
> 2. **LLM output bias:** Does `prompt_builder_service.dart` use a system prompt that biases tone or language? Are Khmer queries forced through English-only prompts?
> 3. **Insights sentiment bias:** In `lib/views/insights/`, does the sentiment chart depend on an English-only classifier? Are tag frequencies normalized across languages?
> 4. **UX defaults:** Are user-affecting defaults set in a way that nudges behavior — cloud sync on by default, anonymous mode hidden, etc.? Check `lib/views/setting/` and `lib/initializer.dart`.
>
> Return only the table format above. Most rows will be `File:Line = N/A` interpretive findings (these go into a "Conceptual findings" section in `00_report.md`). Categories: `RagBias`, `LlmBias`, `UxBias`. Severity: `Medium` typical; `High` if a bias actively excludes Khmer users from a feature. Effort: usually `M`/`L`.

#### 4d. neg-correlation-auditor → `08_neg_correlation.md`

Prompt:

> You are the Trovara negative-correlation auditor. Read-only. Read `CLAUDE.md`, `lib/core/services/ai/embedding_service.dart`, `lib/core/services/sync/`, `lib/core/services/chat/chat_drive_sync_service.dart`, `lib/views/trash/`, `lib/views/insights/`. Then look for:
>
> 1. **Features that conflict:** Embedding cache (in `embedding_service.dart`) invalidation racing with note edits in `note_view_model.dart`. Drive sync overwriting unsynced local edits (compare timestamps in `google_drive_sync_service.dart`). Trash restore breaking embedding linkage (does restoring a note from trash re-embed it, or leave dangling embeddings?).
> 2. **Quality vs. cost trade-offs:** Where does increasing RAG accuracy (top-k, query expansion) directly inflate API cost or latency? Look at `multi_query_expansion_service.dart` and `rag_service.dart`. Note the trade-off without prescribing a value.
> 3. **Insights chart correctness:** Read the sentiment + tag-frequency chart math in `lib/views/insights/`. Does it correctly handle negative correlations (or does it confuse anti-correlation with no correlation)? Are zero-sample buckets handled?
>
> Return only the table format above. Categories: `FeatureConflict`, `Tradeoff`, `MathError`. Most rows will be `N/A` line or specific `file:line` for math errors. Severity: `High` for active data corruption (e.g., sync overwriting edits); `Medium` for trade-offs that should be surfaced to the user; `Low` for math nits. Effort: usually `M`/`L`.

#### 4e. perf-auditor → returned table appended to `05_perf_deps.md` by orchestrator

Prompt:

> You are the Trovara performance auditor. Read-only. Read `CLAUDE.md`, `lib/core/services/ai/embedding_service.dart`, `lib/core/services/ai/embedding_chunker.dart`, `lib/views/search/search_content.dart`, `lib/core/services/notes/note_service.dart`. Then look for:
>
> 1. **Main-isolate work:** Embedding chunking (`embedding_chunker.dart`) running synchronously on the UI isolate. Large markdown-to-Quill conversion (`markdown_to_quill_converter` if exists) blocking UI.
> 2. **Unbounded list rendering:** `search_content.dart` (875 LOC) — is the result list virtualized (ListView.builder) or fully built?
> 3. **ObjectBox query patterns:** Repository methods that load `getAll()` then filter in Dart (should use ObjectBox query conditions).
> 4. **Memory leaks:** Image cache not capped, large notes held in chat history forever, stream subscriptions accumulating.
>
> Return only the table format above (no preamble). Categories: `MainIsolate`, `ListRender`, `QueryPattern`, `MemLeak`. Severity: `High` for confirmed jank (>16ms work on UI thread); `Medium` for sub-optimal patterns; `Low` for nits. Effort: `S`/`M`/`L`.

After perf-auditor returns, orchestrator appends the table to `05_perf_deps.md` under a new `## Perf findings` heading. The earlier deps findings (from step 3) stay intact above it. Because step 3 (parallel batch of 4 promoted agents) fully completes before step 4 starts, there is no race on `05_perf_deps.md`.

### 5. Aggregate

After all dispatched agents return:

1. Parse each per-dimension file's table into a flat list of finding objects with fields `{severity, category, file, line, finding, recommendation, effort, source_agent}`.

2. **Dedup:** key by `(file, line ±3 lines)`. On collision: keep higher severity, merge `category` (comma-join), merge `recommendation` (semicolon-join), list both source agents in a `Sources:` column.

3. **Rank:** sort by severity weight first, then by ROI score = severity_weight / effort_weight.

   | Severity | Weight | | Effort | Weight |
   |---|---|---|---|---|
   | Critical | 8 | | S | 1 |
   | High | 4 | | M | 3 |
   | Medium | 2 | | L | 8 |
   | Low | 1 | | | |

   Worked: `Critical/S` (8.0) > `Critical/M` (2.67) > `High/S` (4.0) > `High/M` (1.33) > `Medium/S` (2.0).

4. **Write `00_report.md`** with this structure:

   ```markdown
   # Quality Audit Report — <today>

   ## Summary

   | Severity | Count |
   |---|---|
   | Critical | N |
   | High | N |
   | Medium | N |
   | Low | N |
   | **Total** | N |

   ## Top 10 by ROI

   <table of top 10 ranked findings>

   ## Conceptual findings (no file:line)

   <table of all N/A rows, grouped by source agent>

   ## Full ranked findings

   <table of all dedup'd findings, sorted by score>

   ## Per-dimension drill-downs

   - [Security](01_security.md)
   - [Bugs & crashes](02_bugs.md)
   - [Test gaps](03_test_gaps.md)
   - [Architecture](04_architecture.md)
   - [Perf & deps](05_perf_deps.md)
   - [Fraud](06_fraud.md)
   - [Bias](07_bias.md)
   - [Negative correlation](08_neg_correlation.md)

   ## Remediation plan

   See [09_remediation_plan.md](09_remediation_plan.md).
   ```

5. **Write `09_remediation_plan.md`** with proposed PR batches:

   ```markdown
   # Remediation Plan — <today>

   Proposed PR batches, ordered by urgency and tractability.

   ## Batch 1 — Critical / High with S effort (ship this week)

   <table of all Critical+High findings with effort=S>

   PR scope: one PR with all these fixes. Estimated 1-2 days.

   ## Batch 2 — Critical / High with M effort (ship next sprint)

   <table>

   PR scope: one PR per category (security / bugs / architecture). Estimated 1 week.

   ## Batch 3 — Medium with S/M effort

   <table>

   PR scope: bundle by area. Run `style-reviewer` agent on the diff before opening.

   ## Batch 4 — L-effort items (design needed)

   <table>

   These need a design discussion first. Open a GitHub issue per item before starting work.

   ## Low-priority backlog

   Remaining `Low` findings. Address opportunistically when touching the area.
   ```

### 6. Report back to user

Print:
```
Audit complete.

Folder: docs/superpowers/audits/<today>/
Files written:
  00_report.md           <N findings, top-10 ranked>
  01_security.md         <count>
  02_bugs.md             <count>          (--full only)
  03_test_gaps.md        <count>
  04_architecture.md     <count>
  05_perf_deps.md        <count>          (perf appended if --full)
  06_fraud.md            <count>          (--full only)
  07_bias.md             <count>          (--full only)
  08_neg_correlation.md  <count>          (--full only)
  09_remediation_plan.md <N batches>

Next: review 09_remediation_plan.md and approve Batch 1.
```

## Notes

- All sub-agents are read-only. The orchestrator only writes to `docs/superpowers/audits/<today>/`.
- Default mode is fast (~5 min, 4 parallel agents). `--full` adds ~10 min for the 5 one-shot agents.
- The `style-reviewer` and `test-writer` subagents are NOT invoked here — they are feature-scoped and require human dispatch. This orchestrator targets whole-codebase drift.
- The audit folder is committed to git so trends are visible over time.
